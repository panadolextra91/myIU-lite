<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| git repo → public/history | Committed files are permanent; a leaked secret in history is a disclosure even if later removed |
| `.env` (local/CI) → process env | Secrets cross into the running process at boot via config.Load() |
| seed migration → database | The bootstrap admin credential is written as data; its strength depends on the hash, not the plaintext |
| dev machine → CI runner | Go toolchain version drift (1.25 local vs 1.24 pinned) can cause code to pass locally but behave differently in CI |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | Weak bootstrap admin password `123456` | mitigate | Seed ONLY a bcrypt cost=12 hash literal (never plaintext) in `000002_seed_bootstrap_admin.up.sql`; `must_change_password=TRUE` forces rotation on Phase 2 first login. Acceptance: stored hash starts `$2a$12$`, verifies `123456`, flag is TRUE. (D-07) |
| T-01-02 | Information Disclosure | Secrets in source / git | mitigate | DB URL, JWT secret, Cloudinary creds load from `.env` via caarlos0/env; `.gitignore` excludes `backend/.env`; only `.env.example` (placeholders) committed. Acceptance: `.gitignore` contains `.env`, `backend/.env` untracked, no secret literal in committed source. (criteria #2) |
| T-01-03 | Tampering | Plaintext seed credential echoed into committed files | accept (low) | The literal `123456` appears only as the documented seed value in the seed migration comment / regeneration note; the stored credential is the hash. Documented as dev-only, rotated on first login. |
| T-01-SC | Tampering | go/npm package installs | mitigate | All packages are pre-committed in CLAUDE.md and audited [Approved] in RESEARCH "Package Legitimacy Audit" (no [ASSUMED]/[SUS]/[SLOP]). No new packages introduced — no legitimacy checkpoint required this plan. |
