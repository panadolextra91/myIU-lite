<objective>
Deliver the forced first-login change-password vertical slice end-to-end: a logged-in user changes their own password (verifying the current one, length 6+, must differ), the session ends and both cookies clear forcing re-login, and — critically — the server-side allow-list (D-15) restricts a must_change_password user to exactly three endpoints so bypassing the SPA does not unlock anything else. The frontend forces such a user onto /change-password until they reset.

Purpose: This is ROADMAP existential pitfall #1 — forced reset enforced server-side, not by the SPA (success criterion 3). It also establishes the self-ownership pattern (user_id from JWT, never from body — AUTH-05) and the session-kill-on-change behavior via password_changed_at (D-13/D-16). The bootstrap admin (admin/123456, must_change_password=TRUE) is the live fixture: it hits forced reset on first login.

Output: POST /auth/change-password endpoint; AuthMiddleware step 5 (allow-list); FE ChangePassword page + post-login redirect. AUTH-03 (change own password) and AUTH-04 (server-side must_change_password restriction) land here.

## Phase Goal

**As a** user logging in with a default/admin-reset password, **I want to** be forced to change my password before I can do anything else, **so that** no account keeps running on a known default credential.
</objective>
