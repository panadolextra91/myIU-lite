<objective>
Retire the existential Cloudinary risk (RESEARCH Assumption A1) and deliver the first end-to-end assignment slice: a lecturer creates an assignment (deadline + late policy), a student submits a magic-byte-validated PDF/ZIP (<=10MB) stored on Cloudinary as an authenticated asset, the server enforces the late policy by its own clock, and downloads are served only through backend-minted short-lived signed URLs gated by ownership.

This plan also lays the single migration 000006 that creates ALL EIGHT phase tables at once (D-06: one incremental migration per phase) — later slices (grading/notifications, quiz authoring, quiz take) consume these tables but MUST NOT add their own migration.

Purpose: ASMT-01..05 are the highest-risk requirements in this phase (first Cloudinary integration + the upload-validation and authenticated-delivery existential pitfalls). Building and proving them first means nothing downstream rests on an unproven storage path.

Output: `internal/shared/cloudinary/` client (+ spike test), migration 000006 (8 tables), `internal/assignments/` feature folder, assignment + submission sqlc queries, frontend coursework API client, lecturer + student assignment pages, route wiring.
</objective>

<phase_goal>
**As a** lecturer or enrolled student, **I want to** post file-upload assignments and submit PDF/ZIP work that is safely stored and served, **so that** coursework can be exchanged without email and without exposing files to the public internet.
</phase_goal>
