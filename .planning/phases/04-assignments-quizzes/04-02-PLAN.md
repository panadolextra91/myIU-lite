<objective>
Deliver the assignment grading slice and the shared notification primitive it fires. A lecturer grades a student's active submission; in the SAME transaction the grade is persisted and a fully-rendered notification row is created for the student (NOTIF-02, D-55). The student sees an unread badge on a header bell, a notification list page, mark-read-on-click, and deep-link navigation (D-54). This notification primitive (table + feature folder + bell) is the shared substrate Phase 5 reuses.

Purpose: ASMT-06 closes the assignment loop (grade -> student is notified) and stands up NOTIF-01/NOTIF-02 — the one-row-per-recipient persisted notification with same-transaction write — which is the architectural anchor for all future notification events.

Output: `internal/notifications/` feature folder, notification sqlc queries, the same-transaction grade+notify path in `internal/assignments/service.go`, the grade route + grading form, `NotificationBell` header component, the notifications list page, route + header wiring.
</objective>

| T-04-09 | Information Disclosure (IDOR) | notification list / mark-read (NOTIF-01) | mitigate | `recipient_id` from JWT for list/unread-count; `MarkRead` WHERE guards `recipient_id = jwt user_id`. Verify: a user cannot mark another user's notification read. (Task 1) |
| T-04-10 | Tampering | notification content drift (D-53) | accept | Title/body persisted rendered at creation; if the resource later changes, the notification text is intentionally stable (historical record). No mitigation needed; documented design choice. |
| T-04-11 | Information Disclosure | over-notification (D-55) | mitigate | Quiz grading deliberately emits NO notification; only assignment grading does. Verify: zero `InsertNotification` references in the quizzes feature. (Task 2) |
</threat_model>
