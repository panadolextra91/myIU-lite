<objective>
Deliver the quiz authoring slice: a lecturer assigned to a course creates a quiz with the full D-03 config (pool size N, max questions M, max grade, shuffle, retake count) plus the D-49 availability window (open-at/close-at), then populates the question pool either by uploading the fixed-format CSV (`question,A,B,C,D,correct`) or by entering single-choice and multi-choice questions in the UI (D-48). Options are stored with stable IDENTITY ids — the correct-answer mapping key that the take-quiz slice (Plan 04) shuffles around (QUIZ-04). The student-facing serializer (`StudentOptionView`) is introduced here with NO `is_correct` field so answer-key leakage is a compile-time impossibility (QUIZ-03 boundary, consumed in Plan 04).

Purpose: QUIZ-01/QUIZ-02/QUIZ-04 (authoring side) build the question-bank substrate that the take/grade slice depends on. Establishing the `StudentOptionView` DTO boundary now means the take endpoint cannot leak correct answers later.

Output: `internal/quizzes/` feature folder (authoring half), quiz + question/option sqlc queries, the lecturer quiz authoring page, route wiring.
</objective>

<phase_goal>
**As a** lecturer, **I want to** build a quiz from a reusable question pool via CSV or the UI, **so that** each student attempt can draw a fresh randomized set without me writing fixed papers.
</phase_goal>
