<objective>
Deliver the student take-quiz slice and the attempt state machine — the security-critical heart of the quiz feature. A student starts an attempt (consuming one retake, drawing M random questions from the pool of N, shuffled if configured), resumes an in-progress attempt without consuming another retake, submits to trigger idempotent auto-grading (exact-match: single == correct, multi set-equality), and reviews results under the window-bound reveal policy. The take API serves only `StudentOptionView` (no `is_correct`); correct answers appear only after `close_at` (D-51, QUIZ-03). An attempt left IN_PROGRESS at window close is lazily auto-submitted (AUTO_SUBMITTED). Retake limits are enforced and the official score is MAX across completed attempts (D-50).

Purpose: QUIZ-03/QUIZ-04 (take side)/QUIZ-05/QUIZ-06 are where the answer-non-leakage existential pitfall is actually enforced at runtime, and where the consume-on-start / resume / idempotent-grade / lazy-auto-submit invariants (D-52) live.

Output: `quiz_attempts` sqlc queries, the attempt state machine in `internal/quizzes/service.go`, take/submit/review routes + the window-bound review DTO, the student take-quiz page, route wiring.
</objective>

<phase_goal>
**As a** student, **I want to** take a quiz with a fresh shuffled question set, submit for an instant score, and review my results, **so that** I get auto-graded feedback without ever seeing the answer key while the quiz is still open.
</phase_goal>
